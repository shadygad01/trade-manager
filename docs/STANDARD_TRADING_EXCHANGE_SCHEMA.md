# Standard Trading Exchange Schema (STES) — v1.1, frozen

The **official, permanent input contract between any external AI and Trade
Manager**. Any AI (ChatGPT, Claude, Gemini, an OCR pipeline, any future
system) that extracts trading information from broker documents fills the
**raw-extraction workbook** defined here; Trade Manager imports it through
the same Import page — and the same Raw Evidence pipeline — as every
screenshot, PDF, and CSV.

- Template (the file AIs actually fill): [`docs/templates/STES-v1.1-raw-extraction-template.xlsx`](templates/STES-v1.1-raw-extraction-template.xlsx)
- Importer: `src/infrastructure/stes/StesWorkbookParser.ts`, routed by `ImportOrchestrator`
- v1.1 is **frozen**. Minor revisions may only ever add optional columns or
  enum values (any 1.x file imports on any 1.y importer; unknown extra
  columns are ignored); semantics never change without a major version, and
  no major version is planned.

## What the workbook is — and is not

Trade Manager is an **Evidence Investigation Engine**. The workbook carries
**raw observations only**: each row records that one source document mentions
one execution, exactly as printed there. It is *not yet* the truth, *not yet*
transactions, *not yet* the ledger, *not yet* holdings.

Therefore the workbook contains **zero matching, zero verification, zero
deduplication, zero ledger logic**:

- The same real-world execution appearing in three documents (statement +
  invoice + orders screen) is **three rows** — one per document, each copying
  exactly what its document shows. The generating AI never suppresses a
  repeat, never picks the "best" source, never merges fields across
  documents. Those duplicate observations are not noise; they are precisely
  what lets the Verification Engine corroborate an execution.
- There is no `Ledger Transaction ID` column — ledger identity is minted by
  Trade Manager at acceptance time and never exists at extraction time.
- There is no sell-allocation sheet — no broker document states *which lots*
  a sell closes; allocation is an explicit in-app decision (never
  FIFO/LIFO/average cost).
- There is no "same execution as" hint — even as a hint it would be a
  matching claim, and cross-document identity resolution is entirely Trade
  Manager's job after import.

All matching, corroboration, contradiction detection, verification,
completeness analysis, and the Evidence Graph run **inside Trade Manager**,
over the total evidence set (which the generating AI cannot see) — never
inside the exchange file.

## Workbook structure

One `.xlsx`, exactly these sheets:

| Sheet | Purpose | Required |
|---|---|---|
| `Metadata` | Self-identification: schema name/version, generator, defaults. | Yes |
| `Documents` | One row per uploaded source document — the evidence-node identity layer. | Yes |
| `Observations` | One row per (document × execution mention). | Yes |
| `Instructions`, `Examples` | Fill guidance shipped in the template; ignored by the importer. | No |

Formatting rules: dates ISO `YYYY-MM-DD`; times 24-hour `HH:MM[:SS]`; plain
decimals (no separators/symbols); enums uppercase; **a blank cell means
unknown — never `0`, `N/A`, or a guess**; never fabricate times (no `00:00`
filler — a real, differing time on two otherwise-identical rows is proof of
two distinct executions, so filler corrupts matching). The importer also
coerces real Excel date/time cells and day serials, so a workbook needs zero
manual editing before upload regardless of how the generator wrote its cells.

### Sheet `Metadata` (`Key` / `Value` rows)

| Key | Required | Validation | Notes |
|---|---|---|---|
| `Schema Name` | Required | Must equal `Trade Manager Standard Trading Exchange` | A workbook without it is rejected as not-STES. |
| `Schema Version` | Required | Major version must be `1` (any `1.x` accepted) | Unknown major → file rejected with a clear message. |
| `Generated By` / `Generated At` | Recommended | — | Audit trail only. |
| `Default Currency` | Optional | ISO 4217 | Defaults to `EGP`. v1.1 importers accept EGP only. |
| `Timezone` | Optional | IANA name | Defaults to `Africa/Cairo`. |
| `Source Description` | Optional | — | Free text. |

### Sheet `Documents` — one row per source document

| Column | Type | Required | Validation / description |
|---|---|---|---|
| `Document ID` | Text | Required | Unique in the file (`DOC-01`, …). A duplicate ID fails the whole file — observation references would be ambiguous. |
| `File Name` | Text | Recommended | The source file's name, for auditing. |
| `Document Type` | Enum | Required | `STATEMENT`, `INVOICE`, `ORDERS_SCREEN`, `NOTIFICATION`, `EMAIL`, `SCREENSHOT`, `CSV_EXPORT`, `PDF`, `OTHER`. Unknown values import as "other document" evidence with a warning — never silently upgraded. |
| `Broker Name` / `Broker Account` | Text | Optional | As printed. |
| `Statement Date` / `Document Date` | Date | Optional | The document's own date(s). |
| `Page Count` | Integer | Optional | — |
| `Extraction Method` | Enum | Optional | `NATIVE_TEXT` or `VISION_OCR` — how the AI read the document. |
| `Document Notes` | Text | Optional | Free text. |

### Sheet `Observations` — one row per (document × execution mention)

| Column | Type | Required | Validation / description |
|---|---|---|---|
| `Observation ID` | Text | Required | Unique in the file (`OBS-0001`, …). Duplicates reject the later row. |
| `Document ID` | Text | Required | Must reference a `Documents` row; an unknown reference rejects the observation. |
| `Transaction Type` | Enum | Required | `BUY`, `SELL`, `DIVIDEND` only. Anything else (deposits, withdrawals, transfers, funds) rejects the row — out-of-scope operations are never coerced. |
| `Ticker` | Text | Required (all types, in practice) | 2–6 uppercase letters/digits. Blank → the row is rejected **by name, with the company name echoed**, so the user fixes the workbook — never guessed. |
| `Company Name` | Text | Recommended | As printed. |
| `ISIN` / `Exchange` | Text | Optional | As printed; `Exchange` defaults to EGX conceptually. |
| `Trade Date` | Date | Required | Execution date (BUY/SELL) or payment date (DIVIDEND). Rows outside the user's tracked date range (older than the configured start year, or future-dated) are excluded with a count — same rule as every OCR parser. |
| `Trade Time` | Time | Optional (BUY/SELL) | Only when printed. Blank = unknown. |
| `Quantity` | Number | Required (BUY/SELL); must be blank (DIVIDEND) | Positive whole shares. A filled value on a DIVIDEND row rejects it as misclassified. |
| `Price` | Number | Required (BUY/SELL); must be blank (DIVIDEND) | Positive per-share price. |
| `Gross Amount` / `Net Amount` | Number | Optional | Only if the document prints them — cross-check redundancy. A `Gross Amount` contradicting `Quantity × Price` (beyond max(0.05, 0.1%)) flags the row for review but still imports it; numbers are never silently "fixed". |
| `Fees` / `Taxes` | Number | Optional | ≥ 0, as printed. Blank = unknown (recorded as unknown, not zero). |
| `Dividend Amount` | Number | Required (DIVIDEND); must be blank (BUY/SELL) | Positive total cash received. A filled value on a BUY/SELL row rejects it as misclassified. |
| `Dividend Per Share` | Number | Optional (DIVIDEND) | As printed. |
| `Currency` | Text | Optional | Blank → EGP. Any non-EGP value rejects the row (v1.1 importers are EGP-only). |
| `Order Type` | Enum | Optional | `LIMIT` / `MARKET`, as printed. |
| `Source Page` | Integer | Optional | Page within the source document. |
| `Transaction Reference` | Text | Optional — the most valuable traceability field | The broker's per-execution identifier, copied exactly. Becomes the fact's `transactionNumber`, the strongest same-execution signal in matching. |
| `Order Number` / `Execution Number` | Text | Optional | As printed. |
| `Extraction Confidence` | Enum | Optional | `HIGH`/`MEDIUM`/`LOW`; blank → MEDIUM. Maps to the fact's `confidence`; a LOW row is never hidden, only visibly cued. |
| `Extraction Notes` | Text | Optional | The AI's honest doubts, shown at review. Legacy compatibility: the exact value `Needs Confirmation` (case-insensitive), with `Order Status` left blank, is still treated the same as `Order Status` containing "partial" (see below) — a workbook generated before `Order Status` existed keeps importing unchanged. |
| `Order Status` | Text | Optional | The broker's own status text, verbatim (`Fulfilled`, `Partially filled`, `Partially filled, canceled`, `Partial fill`, `Cancelled`, …) — see Instructions rules 16–17. Classifies the row into one of the three execution states this app recognizes: **containing "cancel" but not "partial"** → the row is a fully-cancelled order — `Quantity`/`Price` are not required, and the row is imported as an audit-only `CancelledOrder` fact, never a BUY/SELL candidate, never entering the ledger. **Containing "partial"** (with or without "cancel") → a partial-fill execution — imported as a normal BUY/SELL candidate, flagged `needsConfirmation`, held as a `PendingExecution` (not yet a `Trade`/`TradeAllocation`) until a broker invoice confirms its real executed shares/price, surfaced on the Portfolio Detail page's holdings table with an upload action in the meantime. **Blank or "Fulfilled"** → an ordinary candidate, imported and ledgered exactly as before. |

**Validation model**: file-level failures (unreadable workbook, missing
sheet/required column, wrong schema name/major version, duplicate Document
ID) reject the whole file with a specific message. Every row-level problem
rejects **only that observation**, with a warning naming its Observation ID
and the exact reason — a partially-bad workbook is never silently dropped
and never silently completed.

## How an imported observation becomes Raw Evidence

STES is **another input source equivalent to PDF/OCR, not a parallel
system**. `ImportOrchestrator.importFile` routes `.xlsx` files to
`parseStesWorkbook`, which returns the exact `ParsedTradeCandidate` /
`ParsedDividendCandidate` shapes every OCR parser returns. From there the
pipeline is byte-for-byte the existing one:

1. The workbook is archived permanently (`Upload`, with file hash + original
   bytes) — the workbook itself is an evidence document.
2. Every observation is appended as one immutable `RawTransaction`
   (`BuyExecution` / `SellExecution` / `DividendPayment`) via
   `importRecording.ts`, stamped `extractionMethod: "stes-workbook"`,
   `parserVersion`, per-row `confidence`, and — crucially — the **source its
   Documents-sheet type maps to**, so the engines treat an STES-carried
   statement observation identically to a statement this app parsed itself:

   | Document Type | RawTransaction source |
   |---|---|
   | `STATEMENT` | `statement` |
   | `INVOICE` | `invoice` |
   | `ORDERS_SCREEN` | `orders-screen` |
   | `CSV_EXPORT` | `csv` |
   | `NOTIFICATION` | `notification` |
   | `EMAIL` | `email` |
   | `SCREENSHOT` | `screenshot` |
   | `PDF`, `OTHER`, unknown | `other-document` |

   `PDF` deliberately maps to `other-document`, not `statement` — "some PDF"
   says nothing about the document type, and inventing statement-level
   authority for it would be fabricated trust. The four STES-only sources
   rank at the generic tier of the evidence-authority ladder
   (`evidenceAuthority.ts`), below every named document type.
3. The existing engines then run unchanged: duplicate/matching
   (`duplicateDetection.ts` — two observations of one execution are flagged
   as one suspected pair, never auto-removed), cross-source corroboration
   ("two documents agree" verification), the Completeness Engine (missing
   buy-history recovery plans), the Constraint Check, and the Evidence Graph.
4. **Nothing is written to the Ledger automatically.** Observations sit in
   the Import review pool until the user assigns portfolios and explicitly
   confirms — the same two-phase Extract → Verify & Distribute workflow as
   every other document. Only then are ledger facts committed and projections
   regenerated.
5. **Two rows never become a Ledger Entry at all, by construction, not by a
   status check that could be bypassed:**
   - A row whose `Order Status` needs confirmation (see the `Order Status`
     column above) is recorded as a `PendingExecution` instead of a `Trade`/
     `TradeAllocation` — it has no Ledger Entry, no Holdings/cost-basis
     impact, and isn't allocatable until a broker invoice is uploaded and
     matched, at which point the SAME `PendingExecution` row is updated (a
     `Trade` is only created then, for a BUY; a SELL still requires its own
     explicit lot-allocation step, per ADR-002, before it's ledgered).
   - A row whose `Order Status` is a full cancellation is recorded as a
     `RawTransaction` of kind `CancelledOrder` — never a `ParsedTradeCandidate`
     at all, so `commitEngine`/`TradeService`/`computePositions` never read
     it. It's audit trail only.

## Rules for generating AIs (the contract, in one place)

The same rules ship inside the template's `Instructions` sheet:

1. One row per mention of one execution in one document. Three documents
   showing the same execution = three rows. Never merge, never pick a "best"
   source, never combine fields across documents.
2. Never aggregate within a document either: three buys printed = three rows,
   even at the same ticker/day/price.
3. `BUY` / `SELL` / `DIVIDEND` only; skip everything else entirely.
4. Blank = unknown. No `0`/`N/A` filler, no fabricated times, tickers,
   references, or ISINs.
5. Copy identifiers exactly as printed — they are matching keys.
6. List every document in `Documents` first; link every observation by
   `Document ID`.
7. Set `Extraction Confidence` honestly; explain doubts in
   `Extraction Notes`.
8. Do not decide duplicates, do not reconcile, do not allocate sells, do not
   compute positions or P/L. That is Trade Manager's investigation, later.
9. Write the broker's own status text into `Order Status` verbatim. A
   partial fill (`Partially filled`, `Partially filled, canceled`, `Partial
   fill`) is never ignored — extract it like any other BUY/SELL row, never
   estimating the executed quantity. A fully cancelled order (zero shares
   executed) is still recorded — Ticker/Trade Date/BUY-SELL only, `Quantity`/
   `Price` left blank — but is never treated as an execution.
